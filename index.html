Jenkins Webhooks

Webhooks are a common way to trigger Jenkins jobs automatically when code changes or external events occur. They let another service send an HTTP request to Jenkins so a build or pipeline can start without manual intervention.

How Jenkins Webhooks Work

The basic flow is:

  • A source system or service (like GitHub, GitLab, Bitbucket, or a custom app) detects an event.
  • That service sends an HTTP POST request to Jenkins with event data.
  • Jenkins receives the webhook request on a configured endpoint.
  • Jenkins validates the request, then triggers a job or pipeline.

Common Webhook Use Cases

  • Triggering a Jenkins build when code is pushed to a repository.
  • Starting a pipeline after a pull request is opened or updated.
  • Running tests automatically when a branch is updated.
  • Deploying code after a successful merge to a main branch.

Jenkins Webhook Endpoints

Jenkins exposes endpoints that can accept webhook payloads. The most common endpoint is part of the Jenkins Git plugin or the GitHub/GitLab integration:

https://jenkins.example.com/github-webhook/

For generic polling or remote trigger jobs, Jenkins also supports:

https://jenkins.example.com/job/my-job/build?token=MY_SECRET_TOKEN
Tip: Use a secure token or secret to protect your webhook endpoint from unauthorized triggers.

Setting Up GitHub Webhooks for Jenkins

Steps to configure GitHub to trigger Jenkins:

  1. Create or open a Jenkins job/pipeline that has GitHub integration enabled.
  2. Configure the job to accept GitHub webhook events, usually via the Git plugin or GitHub plugin.
  3. In your GitHub repository, go to Settings > Webhooks.
  4. Add a new webhook with the Jenkins URL, typically https://jenkins.example.com/github-webhook/.
  5. Select the events you want to trigger builds, such as Push events or Pull request events.
  6. Save the webhook. GitHub will send a test payload automatically.

Using a Remote Trigger Token

If you do not use a dedicated GitHub/GitLab plugin, you can use Jenkins remote build tokens:

https://jenkins.example.com/job/my-job/build?token=MY_SECRET_TOKEN

Then configure the webhook on your SCM service to POST to that URL when a change occurs.

Webhook Security Best Practices

  • Use HTTPS to encrypt webhook traffic.
  • Require a secret or token to authenticate webhook requests.
  • Limit webhooks to only the events needed for your workflow.
  • Validate payload signatures when supported by the source service.

Jenkins Pipeline Example

In a Jenkins Pipeline, webhook-triggered jobs are often defined with SCM polling disabled and webhook triggers enabled:

pipeline {
    agent any
    triggers {
        githubPush()
    }
    stages {
        stage('Build') {
            steps {
                echo 'Building...'
            }
        }
    }
}

Why Use Webhooks?

Webhooks make Jenkins builds reactive instead of polling-based. This reduces load, improves responsiveness, and makes CI/CD workflows more efficient.

With Jenkins webhooks, you can create fast, event-driven pipelines that respond immediately to repository changes or external events.
